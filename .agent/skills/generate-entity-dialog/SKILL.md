# Generate Entity Dialog Skill

Generate a consistent create/edit dialog pattern for entity management pages that already use a feature data table (`{Entity}DataTable`).

## When to Use

Use this skill when the user asks to:
- Add create/edit dialogs for a list entity page
- Add delete-from-dialog confirmation flow
- Add entity selection + hydrate-on-edit behavior
- Extend a page generated by `generate-data-table` with popup create/edit UX

Primary upcoming targets:
- Filters
- Counties
- Operators
- Buyers (including acquisition-context quick-add)

## Prerequisite

Run after (or alongside) `generate-data-table`. The page should already render a feature table component and expose row edit actions.

## Required Inputs

| Parameter | Required | Description |
|-----------|----------|-------------|
| `entityName` | Yes | PascalCase entity name (`Filter`, `County`, `Operator`, `Buyer`) |
| `entityType` | Yes | Entity TS type |
| `entityTypePath` | Yes | Entity type import path |
| `pagePath` | Yes | Page that hosts table + dialog |
| `dialogComponentPath` | Yes | Path for new dialog component |
| `hookName` | Yes | Hook used by page (`useFilter`, `useCounty`, etc.) |
| `storeSelectionAction` | Yes | Store-backed selection action (`setSelectedX`) |
| `selectedEntityKey` | Yes | Hook value for selected entity (`selectedX`) |
| `selectedEntityLoadingKey` | Yes | Hook value for selected-entity loading (`isSelectedXLoading`) |
| `createAction` | Yes | Hook create action |
| `updateAction` | Yes | Hook update action |
| `deleteAction` | Yes | Hook delete action |
| `idField` | Yes | Entity ID field name (case-sensitive) |
| `nameField` | No | Name/title field used in dialog and uniqueness checks |
| `extraDialogProps` | No | Optional context props (for buyer quick-add, etc.) |

> [!CAUTION]
> Verify identifier case exactly (`viewID` vs `viewId`, `buyerID` vs `buyerId`).

## Output Artifacts

Generate/update these:
1. `{entity}-editor-dialog.tsx` component
2. `page.tsx` integration (or host component integration)
3. Optional hook/store surface extension when missing

## Target Pattern

### 1) Dialog component contract

Use a controlled dialog API:

```ts
interface EntityEditorDialogProps {
  isOpen: boolean;
  onClose: () => void;
  mode: 'create' | 'edit';
  entity?: EntityWithDetails;
  isLoadingEntity?: boolean;
  entities: EntitySummary[];
  onSave: (payload: SavePayload) => Promise<void>;
  onDelete?: () => Promise<void>;
  // optional context props for embedded flows (e.g. buyer quick-add)
}
```

Behavior requirements:
- Initialize form from `entity` when edit data is available.
- Show loading placeholder while `isLoadingEntity` is true.
- Prevent save/delete while loading or submitting.
- Keep validation local and typed (no `any`).
- Use `sonner` toasts for save/delete outcomes.
- Keep Flat 2.0 styling: no borders on container sections.

### 2) Page integration contract

Page responsibilities:
- Keep table + toolbar + dialog open state.
- Keep `dialogMode` and `editingEntityId`.
- On dialog open in edit mode, call store-backed `setSelectedEntity(editingEntityId)`.
- On close, clear selected entity via `setSelectedEntity(null)`.
- Pass `selectedEntity` and `isSelectedEntityLoading` into dialog.
- Use hook/store actions for create/update/delete only.

Required effect shape:

```ts
useEffect(() => {
  if (!showDialog) return;

  if (editingEntityId && dialogMode === 'edit') {
    setSelectedEntity(editingEntityId).catch(handleError);
    return;
  }

  setSelectedEntity(null).catch(handleError);
}, [showDialog, editingEntityId, dialogMode, setSelectedEntity]);
```

### 3) Store/hook expectations

If missing, add these to the entity hook/store surface:
- `selectedEntity: EntityWithDetails | null`
- `isSelectedEntityLoading: boolean`
- `setSelectedEntity(id: string | null): Promise<void>`

Data-loading rules:
- Hydrate selected entity details only once per entity ID.
- Reuse cached details on repeated edit opens.
- Do not fetch repeatedly on every render/open cycle.
- Keep API communication in store actions, not components.

## Buyer-specific extension (Acquisitions context)

For buyer quick-add popup flows from acquisition editing:
- Allow dialog to accept optional context payload (e.g. callback with created buyer ID/name).
- Return created buyer to caller without forcing full-page navigation.
- Keep core dialog reusable for both standalone Buyer management and acquisition-embedded quick-add.

## Generation Steps

### Step 1: Create dialog component

Create `<entity>-editor-dialog.tsx` in feature component folder.

Minimum capabilities:
- create mode + edit mode
- delete confirmation (edit mode)
- loading state for edit hydration
- local validation + disabled action states
- `data-testid` on interactive controls
- **DialogDescription for accessibility** (required for screen readers)

**Accessibility Requirements:**
- Import `DialogDescription` from `@/components/ui/dialog`
- Include `DialogDescription` in both loading and main dialog states
- Loading state: Add `DialogHeader` with `DialogTitle` and `DialogDescription`
- Main dialog: Add contextual description that changes based on mode (create/edit)

Example structure:
```tsx
import {
  Dialog,
  DialogContent,
  DialogDescription,
  DialogHeader,
  DialogTitle,
  DialogFooter,
} from '@/components/ui/dialog';

// Loading state
<DialogContent>
  <DialogHeader>
    <DialogTitle>Loading...</DialogTitle>
    <DialogDescription>Please wait while we load the {entity} details.</DialogDescription>
  </DialogHeader>
  <div className="flex items-center justify-center py-8">
    <div className="animate-spin rounded-full h-8 w-8 border-b-2 border-primary" />
  </div>
</DialogContent>

// Main dialog
<DialogHeader>
  <DialogTitle>
    {isEditMode ? 'Edit {Entity}' : 'Create {Entity}'}
  </DialogTitle>
  <DialogDescription>
    {isEditMode ? 'Update the {entity} information below.' : 'Enter the details for the new {entity}.'}
  </DialogDescription>
</DialogHeader>
```


### Step 2: Integrate dialog into page

Update page to:
- open dialog in create mode from toolbar button
- open dialog in edit mode from row action/double click
- wire `handleSave` to create/update
- wire `handleDelete` to delete action
- wire selection hydration effect and clear-on-close behavior

### Step 3: Optional store/hook augmentation

If the entity hook/store does not expose selected-entity hydration:
- add selected-entity state + loading flag
- add one-time cached hydration action
- expose via custom hook

### Step 4: Verify

```bash
pnpm eslint <dialog-file> <page-file> <hook-file> <store-file>
pnpm exec tsc --noEmit
```

If repository baseline is clean:

```bash
pnpm lint
```

## Architecture Rules

- Components must use hooks only; no direct store import in page/component code.
- Hooks/components do not call API clients directly.
- Store methods called frequently must prefer cached data over repeated API calls.
- No `any` types.
- File size max 500 lines; split into folder modules when needed.
- Flat 2.0 design rules are mandatory.
- Use `pnpm` only.
